System Overview
Maritime Network Management System (NMS)

The Maritime NMS is a comprehensive solution for monitoring and managing maritime vessel systems in compliance with the Korean Register's "Guidance for Cyber Resilience of Ships and Systems" (GC-44-K, 2024), which incorporates IACS UR E26 and E27 requirements.

Core Functionality
  • Vessel Asset Inventory: Comprehensive tracking of hardware and software components across all onboard Computer Based Systems (CBS).
  • Security Zone Management: Implement and visualize security zones for proper network segmentation.
  • Real-time Monitoring: Monitor system performance, network traffic, and security status.
  • Alert Management: Generate, track, and respond to security and operational alerts.
  • Security Log Management: Maintain comprehensive logs for audit and incident response.
  • User Access Control: Role-based access control with least privilege principles.
Technical Architecture
  • Backend: Python Flask web framework
  • Database: PostgreSQL with SQLAlchemy ORM
  • Authentication: Flask-Login with role-based permissions
  • Frontend: HTML5, CSS3, JavaScript with Bootstrap 5
  • Data Visualization: Chart.js for dashboards, D3.js for network visualization
  • API: RESTful API endpoints for data collection and retrieval

User Guide
Dashboard

The Dashboard provides a real-time overview of all vessels, critical alerts, and system status. Key features include:

  • Alert counters by severity
  • Vessel summary cards with security zone information
  • Recent alerts list
  • System performance graphs
  • Latest sensor readings
Assets Management

The Assets page allows viewing and managing all Computer Based Systems (CBS) assets across vessels:

  • Filter assets by vessel, security zone, and asset type
  • View detailed asset information
  • Add new hardware and software assets
  • Track asset relationships (hardware-software)
  • Manage asset firmware and software versions
Security Zones

Security Zones page visualizes and manages the logical security zones across vessels:

  • Zone and Conduit diagrams (KR GC-44-K, 2.402.1)
  • Network diagrams showing data flows between zones
  • Risk level visualization
  • Asset allocation to security zones
Alerts

The Alerts page provides tools for managing security and operational alerts:

  • Filter alerts by status, severity, and vessel
  • View alert details and recommended actions
  • Acknowledge and resolve alerts
  • Track alert history
Security Logs

Security Logs page displays comprehensive audit logs for security events:

  • Filter logs by date range, event type, and vessel
  • View logs in table or timeline format
  • Export logs for compliance reporting
Administration

The Admin panel (for administrators only) provides system configuration options:

  • User management
  • Vessel registration
  • System configuration
  • Backup and recovery
  • Security controls
API Documentation
API Overview

The Maritime NMS provides RESTful API endpoints for data collection and retrieval. These APIs enable integration with vessel systems, sensor data collection, and third-party applications.

Authentication

All API endpoints require authentication using either:

  • API token authentication
  • Session-based authentication (for browser-based access)

API tokens can be generated and managed in the Admin panel.

Key Endpoints
Endpoint Method Description Authentication
/api/sensor_data POST Submit sensor data from vessel systems API Token
/api/vessel_data GET Retrieve vessel data for dashboard User Session
/api/vessel/{id}/security_zones GET Get security zones for a specific vessel User Session
/api/alerts/stats GET Get alert statistics for dashboard User Session
/api/admin/create_user POST Create a new user (admin only) User Session (Admin)

All API endpoints return appropriate HTTP status codes and error messages:

  • 400: Bad Request (invalid data)

  • 403: Forbidden (insufficient permissions)

Security Controls
Authentication and Access Control

The Maritime NMS implements multi-layered security controls to protect against cyber threats:

  • Role-Based Access Control: Users are assigned specific roles (Administrator, Operator, Read-Only) with appropriate permissions.
  • Password Policy: Enforces strong password requirements including minimum length, complexity, and expiration.
  • Multi-Factor Authentication (MFA): Optional MFA can be enabled for users, and is recommended for administrative accounts.
  • Session Management: Automatic timeout of inactive sessions.
  • Failed Login Protection: Account lockout after multiple failed login attempts.
Data Protection
  • Data Encryption: All sensitive data is encrypted in transit using TLS/SSL.
  • Password Security: Passwords are never stored in plain text, only as secure hashes.
  • Integrity Verification: Data integrity checks using cryptographic hashing.
  • Backup Encryption: System backups can be encrypted to protect sensitive information.
Network Security
  • Security Zones: Implementation of logical security zones with appropriate conduits.
  • API Security: Token-based authentication and encryption for API communications.
  • Rate Limiting: Protection against brute-force and denial-of-service attacks.
Logging and Monitoring
  • Comprehensive Logging: All security-relevant events are logged with timestamps.
  • Security Alerting: Real-time alerts for security violations and suspicious activities.
  • Audit Trail: Detailed audit logs for compliance reporting and incident investigation.
Defense in Depth

In accordance with KR GC-44-K, 2.402.4, the system implements a defense-in-depth strategy:

  1. Multiple layers of security controls
  2. Principle of least privilege for user access
  3. Security hardening of system components
  4. Regular security updates and patches
  5. Backup and recovery mechanisms
Incident Response
Incident Response Plan

The Incident Response Plan outlines the procedures for responding to cybersecurity incidents, in accordance with KR GC-44-K, 2.404.1.

Incident Classification
Severity Description Response Time
Critical Severe impact to safety or critical operations Immediate (within minutes)
High Significant impact to operations Urgent (within 1 hour)
Medium Moderate impact to operations High priority (within 4 hours)
Low Limited impact to operations Normal (within 24 hours)
Incident Response Workflow
  1. Detection: Incident is detected via system alerts, logs, or user reports.
  2. Triage: Initial assessment to determine severity and impact.
  3. Containment: Immediate actions to limit the impact of the incident.
  4. Investigation: Detailed analysis to determine the cause and extent of the incident.
  5. Eradication: Remove the cause of the incident.
  6. Recovery: Restore affected systems to normal operation.
  7. Post-Incident Analysis: Review and document lessons learned.
Fallback to Minimum Risk State (KR GC-44-K, 2.404.4)

If a cyber incident impairs the ability of a CBS or network to provide intended services, the system should fallback to a minimum risk state:

  • Identify critical services that must be maintained
  • Define minimum risk states for each critical system
  • Implement fallback procedures to maintain safe operation
  • Test fallback procedures during regular drills
Important: All cybersecurity incidents must be reported to the designated Cybersecurity Officer and documented according to the vessel's incident reporting procedures.

  • Software update history

These reports can be used during audits and inspections to demonstrate compliance with the applicable requirements.
